# SmartRoute 本地智能路线规划系统比赛方案

版本：V1.0  
项目定位：美团场景内的小团路线规划子 Agent  
适用材料：比赛方案文档 / 答辩讲稿基础稿 / Word 或 PDF 提交稿

## 1. 一句话方案

SmartRoute 是嵌入美团 App 场景的小团路线规划子 Agent，将用户自然语言里的出行目标、POI 数据、UGC 口碑、用户画像和时空约束，转化为可直接执行、可实时调整、可解释的本地生活路线。

它解决的不是“推荐几个地点”，而是把“我想在某个区域吃喝玩乐几个小时”变成“现在从哪里开始、下一站去哪、几点到、人均多少、排队多久、如果不满意怎么局部调整”的完整执行方案。

## 2. 任务理解与用户痛点

用户在短途出行、周末游玩、饭前饭后安排、朋友聚会或亲子出行时，通常需要同时处理多个决策：

- 去哪里：先找餐饮、咖啡、展览、娱乐、购物等多个 POI。
- 怎么串：判断这些 POI 是否顺路，是否能在 2-4 小时内完成。
- 怎么取舍：在评分、距离、预算、排队、营业时间和同行人偏好之间反复权衡。
- 怎么改：路线生成后，用户经常会继续说“不要咖啡”“少走路”“换一家餐厅”“不想去这个点”，这要求系统能基于当前路线做局部调整，而不是重新开始。

现有搜索和推荐体验擅长解决单点 POI 决策，但用户仍要自己组合路线；地图产品擅长导航，但不理解 UGC 口碑、排队、预算和美团交易语境；通用大模型能给文本建议，但缺少真实 POI、履约入口和约束校验。SmartRoute 的核心价值就是把“大模型理解能力”和“本地生活数据/工具链”合在一起，降低用户从发现到执行的决策成本。

## 3. 解决方案总览

SmartRoute 采用“美团场景入口 + SmartRoute 插件能力”的产品形态，不做独立 App 首页，而是模拟真实美团使用链路。当前 Demo 覆盖四类入口：

| 入口 | 用户上下文 | SmartRoute 触发方式 | 体现价值 |
|---|---|---|---|
| 搜索页 | 用户搜索区域、商圈、品类或关键词 | `/api/search-preview` 先召回真实候选 POI，用户勾选后智能排路线 | 未知具体 POI 也能从搜索结果进入路线 |
| 问小团 | 用户自然语言提出区域、时间和活动目标 | `/api/route-intent` 判断是否调起 SmartRoute；缺槽时追问 | 小团保持通用助手定位，SmartRoute 作为垂直子 Agent |
| 收藏夹 | 用户已收藏多个店或景点 | 已选 POI 作为 pinned stops，路线补齐并排序 | 把用户已有选择串成可执行路线 |
| POI 详情页 | 用户正在浏览某个商户 | 传入坐标和 `fixed_start_poi_id`，固定当前店为第 1 站 | 从单点商户延展成半日或一晚路线 |

四个入口最终进入同一个路线规划能力：解析用户意图，召回候选 POI，结合画像与 UGC 加权，生成多方案路线，再支持自然语言局部调整和反馈记忆。

## 4. 核心技术链路

```text
用户输入 / 入口上下文
  -> RouteIntentRouterAgent：判断是否调起 SmartRoute，低置信时追问
  -> IntentParserAgent：解析地点、时间、预算、排队、活动类型、同行人等槽位
  -> RouteContext Resolver：融合搜索页、收藏夹、详情页、小团上下文
  -> AMapClient / POIRetrieverAgent：真实 POI 召回或本地 RAG 兜底
  -> 用户画像与 UGC 加权：搜索偏好、收藏、浏览标签、预算、排队容忍度
  -> RoutePlannerAgent：多约束路线生成，保证 >=3 POI 和餐饮 + 文化/娱乐覆盖
  -> AMap Direction Enrichment：补充路线 polyline、步行/公交/打车分段信息
  -> RouteInsight / Tool Trace：输出可解释指标、冲突、风险与规划过程
  -> /api/adjust：基于当前路线做局部调整
  -> /api/feedback：记录喜欢/不合适，沉淀偏好
```

系统采用 LLM 优先、规则兜底的稳健策略。配置 DeepSeek Key 时，意图解析和调整工具选择优先由 LLM 完成；无 Key、超时或返回异常时，本地规则继续保证 Demo 可用。对于明确城市、地标、商圈、收藏 POI 或详情页坐标的请求，系统进入真实地点模式，优先使用高德 Web 服务解析锚点、召回真实 POI 并计算路径；高德失败时返回可解释 trace，不再跨城回退到错误城市的本地 RAG。

## 5. 核心能力

### 5.1 路线生成：从目标到可执行行程

用户可以输入“我下午要去深圳大学附近玩 3 个小时，帮我规划一个路线”，系统会解析地点、时间和活动目标，围绕真实锚点召回候选 POI，输出包含站点顺序、到达/离开时间、停留时长、人均预算、等待时间和交通时间的完整路线。

当前主路线稳定满足比赛硬性约束：

- 路线生成目标耗时 <10 秒。
- 至少串联 3 个 POI。
- 至少覆盖餐饮/咖啡 + 娱乐/文化/散步两类体验。
- 支持 2 条差异化路线方案，便于用户对比。

### 5.2 多条件约束与个性化

路线规划同时考虑多类约束：

- 时间：总时长、开始时间、单站停留和移动时间。
- 空间：城市、区域、锚点、已选 POI、详情页固定起点。
- 预算：人均预算、优惠敏感度、价格偏好。
- 排队：用户最大等待容忍度、低排队路线偏好。
- 活动：餐饮、咖啡、文化、展览、娱乐、购物等类型覆盖。
- 同行人：朋友、情侣、亲子、带父母、少走路等偏好。

系统内置三类模拟画像，并支持脱敏画像导入和评委即时画像：低排队务实型、文艺体验型、带爸妈轻松型。不同画像会影响 POI 召回权重和排序结果，前端会展示“画像信号 -> 召回加权 -> 路线变化”，让用户理解为什么同一个 prompt 会得到不同路线。

### 5.3 实时动态调整：不是清空重规划

比赛要求强调“需要实时互动，而不是重新进行路线规划”。SmartRoute 的 `/api/adjust` 基于当前路线、当前 query 和当前 `route_context` 做局部调整，返回调整前后指标变化，而不是把用户重新带回第一步。

支持的典型指令包括：

- “不要咖啡”“不想喝咖啡”：识别负向类别并替换相关站点。
- “少走路”：调整顺序或替换为更近 POI。
- “换一家餐厅”“想吃粤菜”：搜索调整专用候选并替换餐饮站点。
- “不想排队”：优先替换高等待站点。
- “便宜点”：优先替换高价餐饮/咖啡站点。

调整结果会明确返回 `applied / partial / not_applied`，避免系统无法改善时假装成功；同时展示等待、人均、移动时间、变更站点和建议放宽的约束。

### 5.4 反馈闭环与可解释性

系统支持“喜欢 / 不合适”反馈，写入 SQLite 用户画像，后续路线召回和排序会调整权重。每次路线生成和调整都会返回 `tool_trace`，包括 ParseIntent、BuildSessionProfile、SearchPOI、PlanRoute、MapDirections、ParseAdjustment、SearchReplacementPOI、ValidateConstraints、UpdateRoute、ExplainChanges 等步骤，便于评委看到多 Agent 和 ToolUse 链路。

## 6. 当前接口能力

| 接口 | 作用 | 评审价值 |
|---|---|---|
| `POST /api/route-intent` | 判断小团自然语言是否调起 SmartRoute，支持缺槽追问和多轮补齐 | 体现自然语言入口和子 Agent 调度 |
| `POST /api/search-preview` | 搜索页先召回候选 POI，返回默认勾选项和可传入规划的 `route_context` | 体现搜索 + 路线规划融合 |
| `POST /api/plan` | 生成多 POI 路线，返回候选、路线、指标、画像影响、冲突解释和 tool trace | 体现端到端路线生成 |
| `POST /api/adjust` | 基于当前路线做自然语言局部调整，返回状态、变更、指标 delta 和 trace | 体现实时互动，不清空重规划 |
| `POST /api/replace` | 针对某个站点返回同类替换 POI 和影响说明 | 体现可控替换和履约前修改 |
| `POST /api/feedback` | 记录喜欢/不合适，沉淀到用户画像 | 体现反馈闭环和长期个性化 |

其中 `RouteContext` 是多入口统一的上下文载体，包含 `source`、`city_hint`、`anchor_text`、`anchor_location`、`selected_pois`、`transport_strategy`、`fixed_start_poi_id` 和 `pinned_policy`。搜索页采用 soft pinned 策略，收藏夹采用 pinned stops，POI 详情页采用 fixed start 策略，保证不同入口都能复用同一套规划链路。

## 7. 评审标准映射

| 评审维度 | SmartRoute 对应能力 | 证明方式 |
|---|---|---|
| 创新性 | 小团通用 AI + SmartRoute 垂直子 Agent；LLM 意图识别 + POI 搜索 + 多约束规划 + ToolUse 调整 | 展示问小团自动判断是否调起插件，以及 `/api/adjust` 的 ReAct/ToolUse trace |
| 完整性 | 覆盖搜索页、问小团、收藏夹、POI 详情页四个入口；后端 API、地图、画像、反馈、调整链路完整 | Web Demo 左侧场景切换，中间手机模拟器，右侧 Agent 解释面板 |
| 应用效果 | 路线 >=3 POI，覆盖餐饮 + 文化/娱乐，显示时间、人均、等待、交通和地图路径；支持 <10 秒生成目标 | 用深圳大学、外滩、广州永庆坊、gaga 详情页等场景演示 |

## 8. 演示场景与测试样例

### 场景 A：未知 POI 的自然语言规划

输入：  
“我下午要去深圳大学附近玩 3 个小时，帮我规划一个路线”

预期展示：

- 小团或规划入口识别地点、时间和路线意图。
- 召回深圳大学附近真实 POI。
- 生成至少 3 站路线，包含餐饮/咖啡与文化/娱乐/散步点。
- 展示时间轴、地图、总时长、人均、等待时间和交通时间。

### 场景 B：搜索页先发现 POI，再调起路线

输入：  
“广州永庆坊附近逛吃 3 小时”

预期展示：

- 搜索页通过 `/api/search-preview` 先出现 4-8 个候选 POI。
- 用户可以勾选想保留的地点。
- SmartRoute 将勾选结果作为 soft pinned stops，补齐并排序成路线。

### 场景 C：POI 详情页从 gaga 出发

入口：  
POI 详情页 “gaga 金地威新中心店”

预期展示：

- `fixed_start_poi_id` 固定当前商户为第 1 站。
- 系统从当前店附近补齐后续 POI。
- 后续调整时仍保留 gaga 为起点，不会把它排到最后或替换掉。

### 场景 D：路线生成后的实时调整

输入：

- “不要咖啡”
- “少走路”
- “换一家老字号餐厅”

预期展示：

- 不清空原路线。
- 局部替换或重排相关站点。
- 展示 `adjustment_status`、变更站点、等待/预算/移动时间 delta。
- 右侧 Agent 面板展示 ToolUse 调整链路。

### 场景 E：不同画像产生差异化路线

同一 prompt 下切换画像：

- 低排队务实型：更低等待、更低预算、更稳妥。
- 文艺体验型：更偏展览、设计感空间、拍照友好。
- 带爸妈轻松型：更少走路、室内、休息方便、打车友好。

预期展示：路线站点和解释发生明显变化，并能看到画像信号如何影响召回加权。

## 9. 商业价值与扩展

SmartRoute 可以把美团现有的搜索、点评、收藏、地图、团购、订座、排队、打车和用户画像能力串成一个更短的决策链路：

- 对用户：减少反复搜索和筛选，把“想法”直接变成可执行路线。
- 对平台：提升 POI 曝光效率、到店转化、团购/订座/打车等履约入口转化。
- 对商户：让优质商户进入更完整的出行场景，而不是只作为孤立搜索结果出现。
- 对小团 AI：通用助手不用承担所有垂直规划细节，可以通过子 Agent 完成高复杂度任务。

未来在获得官方授权后，可进一步接入真实美团账号画像、搜索历史、收藏、浏览、订单、排队、优惠、订座和交易状态，让路线规划从 Demo 级能力升级为可生产化的本地生活智能决策入口。

## 10. 当前边界与风险说明

- 当前支持模拟画像、脱敏手动导入画像和评委即时画像，不读取真实美团账号、cookie、手机号或订单隐私。
- `official_api` 是预留 adapter，需要授权后才能启用。
- 明确地点模式依赖高德 Web 服务 Key；Key 类型错误、权限不匹配、IP 白名单或额度限制时，系统返回可解释失败 trace，不跨城生成错误路线。
- 本地 RAG 作为无明确地点或早期演示兜底，不用于覆盖真实城市锚点场景。
- 目前重点覆盖半日、一晚、周末短路线，不覆盖复杂多日旅行行程。

## 11. 结论

SmartRoute 的核心突破是把 LLM 的自然语言理解、POI/UGC 的真实本地生活信息、用户画像的个性化偏好和路线规划工具链融合成一个可执行闭环。它不止回答“去哪儿”，而是直接解决“怎么串、怎么走、怎么改、为什么这样安排”的完整问题，符合比赛对创新性、完整性、应用效果和商业价值的综合要求。
