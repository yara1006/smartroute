# SmartRoute PRD

## 1. 产品定位

SmartRoute 是嵌入美团 App 的本地生活路线规划插件 / 子 Agent。它不是一个独立 App，也不是要替代小团 AI，而是在用户产生吃喝玩乐意图后，把美团侧 POI、UGC、用户偏好、排队风险、预算、时间和距离约束组织成一条可执行路线。

一句话定位：

> SmartRoute 把“我想去哪里玩/吃什么”变成“现在怎么走、几点到、花多少钱、哪里可替换”的可执行本地生活路线。

## 2. 核心场景

SmartRoute 覆盖两类入口：

1. 已知 POI 场景：用户在收藏夹、搜索结果或 POI 详情页中已经选中多个店/景点，点击“帮我排路线”，SmartRoute 负责排序、时间安排、约束满足和执行解释。
2. 未知 POI 场景：用户直接进入“问小团”通用对话界面，只知道区域、时间或目标，例如“我下午要去深圳大学附近玩 3 个小时”或“我下午要去外滩玩 3 个小时”。小团先承接自然语言提问，再由路线意图识别 Agent 判断是否调起 SmartRoute；命中后 SmartRoute 根据用户提到的位置解析地图锚点、发现候选 POI，并串联成路线。

可被调用的入口：

- 小团 AI：用户自然语言提出出行/吃喝玩乐需求后，先由小团判断是否属于路线规划意图，再调起 SmartRoute 插件卡片。
- 收藏夹：用户收藏多个店或景点后，一键生成路线。
- 搜索页：用户搜索“深圳大学附近下午怎么玩”“外滩下午茶”“深圳上城夜市”后，系统推荐“智能排路线”。
- POI 详情页：用户从某个店出发，继续添加下一站。

Web 展示形态：

- 最终 Demo 不是独立 App 首页，而是“美团 App 场景模拟器 + SmartRoute 插件弹出效果”。
- Web 左侧展示四个入口场景：搜索页、问小团、收藏夹、POI 详情页。
- Web 中间展示美团手机模拟器，不同入口下出现不同 SmartRoute 触发卡片。
- Web 右侧展示 Agent 解释面板，说明入口来源、输入上下文、规划链路和评分点命中。
- 不同入口最终进入同一个 SmartRoute 插件路线页：地图 + 路线卡 + 时间轴 + 实时调整 + 反馈记忆。

四个入口在产品中的含义：

| 入口 | 用户上下文 | SmartRoute 触发方式 | 重点表达 |
|---|---|---|---|
| 搜索页 | 用户搜索某个区域、品类或关键词 | 搜索结果中出现“智能排路线”卡片 | 未知具体 POI 也能从候选结果中规划 |
| 问小团 | 用户在通用小团对话里提出区域、时间、吃喝玩乐目标 | 路线意图识别 Agent 高置信命中后，小团回复 SmartRoute 插件卡；中置信时先问一句确认 | SmartRoute 是小团 AI 调起的垂直子 Agent，不占用小团首屏默认 prompt |
| 收藏夹 | 用户已收藏多个商户或地点 | 收藏夹顶部出现“一键排路线” | 已知多个 POI 时负责排序和约束满足 |
| POI 详情页 | 用户正在浏览某个商户 | 详情页出现“从这里出发/加入路线” | 从单点 POI 延展成多站路线 |

四个入口都会把真实上下文传给 SmartRoute，而不是只传固定 prompt：

- 搜索页 / 问小团：传入 `city_hint`、`anchor_text`，例如深圳大学、外滩、金地威新中心。
- 收藏夹：传入用户勾选的 2-5 个 POI，作为必须优先保留的 pinned stops。
- POI 详情页：传入当前商户坐标和名称，SmartRoute 从该店附近补齐下一站。
- 后端优先使用高德 Web 服务做地理编码、周边 POI 召回和路径规划；无 Key 或调用失败时，按锚点附近本地兜底，不再回到默认外滩路线。

小团入口的触发逻辑：

- 小团首屏保持通用助手定位，不默认展示“规划路线”prompt。
- 当用户输入同时包含区域/地点、时间窗口、串联意图、出行人群、预算、排队、少走路等信号时，路线意图识别 Agent 判断是否调起 SmartRoute。
- 高置信命中：直接返回 SmartRoute 插件卡，例如“已为你生成 3 小时外滩路线”。
- 中置信命中：小团先用轻量确认引导，例如“要不要把这些地点排成一条路线？”并提供“排路线 / 只看推荐”两个操作。
- 低置信命中：小团继续按普通问答或单点推荐处理，只在结果卡片里提供次级入口“把这些店排成路线”。

## 3. 用户与痛点

目标用户：

- 本地周末出行用户：想把餐饮、展览、咖啡、逛街串起来。
- 游客/短途旅行用户：只知道区域，不熟悉具体 POI。
- 情侣、朋友、亲子、带父母出行用户：需要根据同行人调整路线强度。
- 决策成本高的用户：不想反复筛选、排队、查距离和预算。

当前痛点：

- 美团/点评搜索擅长找到单个 POI，但不会自动排成路线。
- 地图 App 擅长导航，但不理解预算、排队、UGC 口碑和生活偏好。
- 通用大模型可以聊天，但缺少真实 POI 状态、交易入口和可执行路线约束。
- 用户经常在“想吃好”“不想排队”“预算有限”“时间不够”之间反复取舍。

## 4. 竞品与替代方案分析

SmartRoute 的竞品不是单一产品，而是用户当前解决“本地生活路线规划”时可能使用的一组替代方案。

| 类型 | 代表产品 | 用户会怎么用 | 主要不足 | SmartRoute 差异 |
|---|---|---|---|---|
| 通用大模型 | DeepSeek、豆包、Kimi、ChatGPT | 直接问“帮我安排外滩下午路线” | 缺少美团 POI、UGC、实时价格、排队、营业时间、交易入口和个人历史偏好；答案容易停留在文本建议 | 接入美团侧数据和工具，输出可执行路线、替换项和反馈闭环 |
| 美团内搜索/点评 | 美团搜索、大众点评搜索、榜单、收藏夹 | 手动搜店、收藏、比较评分和评价 | 擅长单点 POI 决策，但不会自动排序成路线；用户仍要自己看距离、时间、排队和预算 | 把多个 POI 自动串联，完成从“发现”到“执行”的一步 |
| 地图导航 App | 高德地图、百度地图、腾讯地图 | 查距离、导航、规划交通方式 | 擅长道路和导航，但不理解本地生活偏好、UGC 口碑、预算、排队和美团履约 | 在美团语境中先解决“去哪、怎么组合、是否值得去”，再连接导航 |
| AI 旅行/路线规划 App | 圆周旅迹、Wanderlog、穷游行程助手等 | 导入攻略、做旅行行程、地图化管理地点 | 更偏旅行/攻略和跨天行程，离美团本地到店、团购、订座、排队、用户评价链路较远 | 聚焦美团本地生活和即时出行，适合半日/一晚/周末吃喝玩乐路线 |
| 小团 AI | 美团内 AI 助手 | 对话获取吃喝玩乐建议 | 更像通用入口和助手，不应承担所有垂直规划细节 | SmartRoute 可作为小团 AI 调起的路线规划子 Agent |

SmartRoute 的核心优势：

- 数据优势：可接入美团 POI、UGC 评价、评分、价格、排队、营业时间、收藏和历史反馈。
- 工具优势：可调用搜索、路线规划、替换 POI、领券、订座、打车、导航等美团工具链。
- 执行优势：输出的是包含时间、预算、等待、交通、替换项的路线，而不是泛泛建议。
- 个性化优势：喜欢/不合适反馈沉淀为用户画像，下次生成路线会调整权重。
- 闭环优势：路线可以直接进入美团履约链路，带动到店、团购、订座、打车和收藏。

因此 SmartRoute 更适合定位为美团内的垂直路线规划 Agent：让通用 AI 负责理解和推理，让美团数据与工具负责真实可执行和交易闭环。

## 5. MVP 范围

当前已实现：

- Web Demo 以美团 App 场景模拟器展示：搜索页、问小团、收藏夹、POI 详情页均可调起 SmartRoute。
- 小团入口支持路线意图识别：优先 DeepSeek，失败或无 Key 时规则兜底；可区分 `open_plugin`、`ask_confirm`、`normal_answer`。
- 自然语言输入路线需求。
- 解析时间、预算、人数、排队、区域、偏好和出行风格。
- 使用本地 POI JSON 和本地索引召回候选 POI。
- 主路线稳定支持 >=3 个 POI。
- 主路线覆盖餐饮/咖啡 + 景点/娱乐。
- 生成 2 条差异化路线方案。
- 支持三种模拟美团用户画像：低排队务实型、文艺体验型、带爸妈轻松型。
- 支持脱敏真实画像导入：用户手动整理搜索词、收藏 POI、浏览偏好、预算、排队和步行偏好后导入，用于路线召回和排序加权。
- 支持动态地点规划：问小团/搜索输入“深圳大学附近”“外滩附近”等位置时，路线围绕用户提到的锚点生成。
- 支持收藏夹和 POI 详情页上下文：收藏夹勾选的店铺会被优先保留，POI 详情页从当前商户坐标延展路线。
- 后端接入高德 Web 服务 adapter：配置 `AMAP_WEB_SERVICE_KEY` 后可做地理编码、周边 POI 搜索和路径 polyline；无 Key 时使用锚点兜底。
- 支持自然语言局部调整：少走路、便宜点、不要排队、加晚餐/咖啡/展览。
- 展示“画像信号 → 召回加权 → 路线变化”，让用户知道搜索偏好、收藏偏好、浏览标签、预算和排队容忍度如何影响路线。
- 展示规划耗时、路线完整性、冲突解释和结构化生成后追问。
- 局部调整后展示 `applied / partial / not_applied` 状态、调整前后指标变化、站点变化和失败时的约束放宽建议。
- 展示路线指标：可信度、总时长、人均、等待时间、交通时间。
- 展示高德地图、站点 marker 和路线连线；未配置 Key 时降级为锚点附近本地兜底路线和示意图。
- 展示候选 POI、路线生成过程和约束解释。
- 支持同类 POI 替换，并显示预算/等待/距离影响。
- 支持喜欢/不合适反馈，写入 SQLite 用户画像。
- 支持未知 POI 场景的动态地点规划，例如“我下午要去深圳大学附近玩 3 个小时”。

下一步必须补齐：

- 在获得大赛方或美团内部授权后，将 `official_api` adapter 接入真实美团账号、搜索、收藏、浏览和交易上下文。
- 将规则版 `/api/adjust` 升级为 DeepSeek Function Calling / ToolUse。
- 深化真实高德道路路径规划策略，包括公交/步行/驾车模式选择和更细的分段交通解释。
- 将“追问卡 + 局部调整”进一步做成可解释的多轮 Agent Trace。

暂不做：

- 真实支付、下单、订座、领券。
- 真实美团账号登录。
- 自动登录、抓取 cookie、读取个人账号隐私数据。
- 多日复杂行程。
- 多人实时协作。
- 正式生产级地图路径规划。
- 大规模真实 POI 数据抓取。

## 7. 验收标准

基础验收：

- 输入“我下午要去深圳大学附近玩 3 个小时，帮我规划一个路线”，能返回围绕深圳大学的至少 1 条路线。
- 输入“我下午要去外滩玩 3 个小时，帮我规划一个路线”，能返回围绕外滩的至少 1 条路线，而不是固定深圳或固定默认样例。
- 从收藏夹勾选 2 个以上 POI 后，路线必须包含用户已选 POI，并自动补齐到 3 站以上。
- 路线展示地图、时间轴、人均、等待、总时长和 POI 说明。
- 生成耗时目标 <10 秒。

当前 P1b 验收：

- 每条主路线 >=3 个 POI。
- 每条主路线至少覆盖餐饮 + 娱乐/文化。
- 同一 prompt 切换不同用户画像后，路线有明显差异。
- 用户自然语言调整当前路线时，不重新清空整个规划流程。
- 调整后能看到等待、人均、移动时间等指标变化。
- 调整无法改善时，不能假装成功，必须展示放宽预算、排队、区域或时间的建议。
- 喜欢/不合适反馈能影响下一次路线生成。
